## 4. Binary Integrity

### 4.1 Reproducible Build

The embedded WASM binary is deterministically reproducible:
- Build command: `zig build -Dembedded=true -Doptimize=ReleaseSmall`
- Two consecutive builds from the same source commit produce **identical SHA-256 hashes**
- Build script: `packages/cell-engine/scripts/reproducible-build.sh`

### 4.2 Binary Manifest

`packages/cell-engine/WASM-MANIFEST.json` records:
- `sha256`: cryptographic hash of the WASM binary
- `zigVersion`: compiler version used
- `sourceCommit`: git commit hash of the source
- `sizeBytes`: binary size (29,613 bytes for embedded profile)

### 4.3 Hash Anchoring

At release time, the SHA-256 hash from the manifest is anchored on the BSV blockchain. Devices verify `SHA-256(loaded_wasm) == anchored_hash` before execution. Hash mismatch → refuse to load → alert operators.

## 5. No Configuration Pathway

There is no runtime flag or database setting that disables linearity enforcement in production:

1. `build.zig:52` — The `embedded` option controls whether BSVZ is linked. It does **not** disable linearity enforcement.
2. `pda.zig:40` — `enforcement_enabled: bool` is set by the executor during script execution. In the production execution path, enforcement is always enabled for semantic objects.
3. The Lean model (`proofs/lean/Semantos/PDA.lean`) does not model a "disable linearity" pathway. Any such pathway would violate K1.

---

## 6. Database Irrelevant

The WASM engine reads cell headers **from stack memory only**:
- `linearity.zig:64` — `getLinearity(cell_data: []const u8)` reads from the `cell_data` slice passed in
- `pda.zig:434` — `slotLinearity` reads `self.main_stack[idx]` — in-memory stack data
- No external database query, no network call, no file I/O in the linearity enforcement path
- The linearity class of a cell is determined by the 4 bytes at offset 16 of its header, read from the stack

## 8. Cryptographic Assumptions

The proof system rests on the following idealized axioms (from `proofs/lean/Semantos/CryptoAxioms.lean`):

1. **SHA-256 collision resistance**: `∀ m1 m2, m1 ≠ m2 → sha256(m1) ≠ sha256(m2)`
2. **ECDSA existential unforgeability** (secp256k1): Verification implies key possession
3. **HMAC-SHA-256 PRF security**: Collision-free as a consequence of PRF property

These are standard cryptographic assumptions. If any is broken, the security of the Bitcoin network (and many other systems) is also compromised.

## 9. Limitations

The following are **honest limitations** of this proof system:

1. **Host import trust boundary**: The WASM binary imports host functions (`host_sha256`, `host_ecdsa_verify`, `host_fetch_cell`). The proofs assume these are correctly implemented. A compromised host can return arbitrary values.

2. **Implementation gap**: The Lean proofs operate on an abstract model. The Zig implementation is a separate codebase. The differential tests and mutation testing bridge this gap but do **not** constitute a mechanical proof of equivalence (e.g., via extraction or refinement types).

3. **Hardware and side channels**: The proofs do not model timing attacks, power analysis, fault injection, or microarchitectural side channels. A compromised CPU or memory subsystem can violate any invariant.

4. **BSV availability**: Hash anchoring depends on the BSV blockchain being available and its consensus rules being upheld. If the BSV network is compromised, hash anchoring provides no guarantee.

5. **Social engineering**: No formal system prevents an authorized operator from replacing a legitimate WASM binary with a modified one and re-anchoring the hash. The system detects unauthorized replacement, not authorized betrayal.

6. **Key management**: The proofs assume private keys are kept private. Key compromise is outside the model.

7. **Formal model scope**: The Lean model covers the 2-PDA kernel (stack operations, linearity, opcodes, termination). It does not model the full application layer, networking, persistence, or UI.

8. **TLA+ model finiteness**: TLA+ model checking explores a finite state space. The properties are verified for bounded models (e.g., bounded number of nodes, bounded history length). Unbounded verification would require theorem proving.
