**Branch**: `feat/u2-udp-multicast-pi-mesh`
**Companion brief**: `UDP-DATAGRAM-DISPATCH-BRIEF.md` (executed) + `UDP-MESH-DIRECTION.md` §3
**Status**: dispatcher + multicast + provisioning + standalone `mesh-node` binary shipped; CLI flag + brain integration deferred.

## Wire-layer stance: IPv6-only

The transport is **IPv6 only** (`AF_INET6` socket with `IPV6_V6ONLY = 1`). Group addresses live in `FF15::/16` (transient site-local) — default `ff15::5e:1`, where the `5e` byte is a "SE"mantos convention. IPv4 multicast groups are rejected at config-load time.

Why v6-only:
1. **Aligned with the IPv6 Forum's `Call for IPv6-Only and Blockchain on Agentic AI`** (Prof. Latif Ladid, 2026) — explicitly pairs IPv6-only with BSV Blockchain as the substrate for the Internet of Agents. The skyminer becomes a literal demo of the thesis.
2. **Direct alignment with [`lightwebinc/bitcoin-shard-proxy`](https://github.com/lightwebinc/bitcoin-shard-proxy)**, which is IPv6-only and uses permanent site-local multicast (`FF05::B:*`) for BSV data-plane fanout. Our `FF15::5e:*` namespace is transient and doesn't collide; both can coexist on the same wire on the same Pi.
3. **No NAT / no v4-mapped fallback** — agent-to-agent reachability is the point. Dual-stack is a temporary compromise; the skyminer testbed skips it.

Group-namespace conventions on this branch:

| Range | Owner | Purpose |
|---|---|---|
| `FF05::B:0000`–`FF05::B:7FFF` | bitcoin-shard-proxy | BSV tx data-plane shards (top bits of txid) |
| `FF05::B:FFFA`–`FF05::B:FFFE` | bitcoin-shard-proxy | BSV control plane (headers, beacons, etc.) |
| `FF15::5e:0001`–`FF15::5e:FFFF` | semantos mesh-node | Substrate gossip (HEARTBEAT, CELL_SYNC, TOPIC_BROADCAST) |

```

# 2. Generate a 2-node config (loopback=true so localhost iface delivers)
bun run ../../tools/u2-mesh/gen-identities.ts \
  --count 2 --loopback true --label-prefix smoke \
```

Expected output on each: `TX heartbeat #N` interleaved with
`RX heartbeat from <peer-label> (...)` lines, both nodes showing
`group=ff15::5e:1:47100`. SIGINT shuts cleanly with
`emitted=N received=N-1` (the last TX racing the signal).

## Generating mesh identities for a real cluster

```
bun run tools/u2-mesh/gen-identities.ts \
  --count 8 \
  --group 239.42.42.42 \
  --port 47100 \
  --out-dir ./pi-cluster
```

Produces `pi-cluster/node-01.json` through `node-08.json`. Each file contains:
- `self.cellId` — the node's 32-byte content-addressed identity
- `self.broadcastSecret` — the 32-byte HMAC key THIS node uses to sign its outbound multicasts
- `peers[]` — every OTHER node's `cellId` + their `broadcastSecret` (a copy used for verification)
- `multicast` — IPv6 group + port + hop limit + loopback flag
- `meta.schema` — `u2-mesh-identity/v2`

**Trust model**: any node that knows peer X's `broadcastSecret` can impersonate X for broadcast purposes. The 8-Pi testbed is single-operator (Todd's lab, all Pis trust each other), so this is acceptable. Phase U.3 replaces these symmetric keys with ECDH-derived secrets bound to the contacts cell-DAG — at that point each peer signs with its own private key and no copy needs to leave the device.

For a 2-process localhost smoke test (helpful before flashing anything):

```
bun run tools/u2-mesh/gen-identities.ts \
  --count 2 \
  --loopback true \
  --label-prefix local \
  --out-dir /tmp/local-mesh
```

`loopback true` is essential for both processes on the same host to receive each other's multicast datagrams. On a real LAN, leave it at the default `false`.

## Pi-cluster deployment

### Confirmed hardware in the skyminer enclosure

Verified by photo 2026-05-21:

- **8 × Orange Pi Prime** SBCs. SoC: Allwinner H5 quad-core Cortex-A53 (aarch64). 2 GB DDR3 RAM. microSD boot, Gigabit Ethernet, onboard WiFi antenna.
- **1 × PL-DGMK300** internal router. Cheap 2018 Chinese "smart wired router". Admin web UI at `192.168.0.1`, credentials `admin`/`admin`. DHCP server for the internal LAN segment. Wires the 8 SBCs together over Ethernet.

**Implications**:
- OS is **Armbian** (Allwinner H5 supported; Raspberry Pi OS won't boot — wrong SoC).
- Boot partition is ext4, not FAT — macOS can't mount it natively (use the SSH-and-install path below rather than dropping config into `/boot` from your laptop).
- IPv6 on the cheap router is irrelevant — our `ff15::5e:*` multicast traffic is link-local/site-local and rides plain L2 switching. As long as the router does basic 802.3 switching, gossip works.

### First-Pi bring-up (one card, prove the cross-compile + the wire)

The safe sequence: bake ONE SD card, boot it alongside the laptop, watch them gossip across the real LAN. Validates aarch64 against Cortex-A53 silicon, validates Armbian+systemd integration, validates IPv6 multicast over the skyminer's actual switch — before you commit 7 more cards.

**Step 1 — Cross-compile the binary on your laptop**

```
cd runtime/semantos-brain
zig build mesh-node -Dtarget=aarch64-linux-gnu
file zig-out/bin/mesh-node
# → ELF 64-bit LSB executable, ARM aarch64, statically linked
```

**Step 2 — Generate a 2-node config on your laptop** (laptop + first Pi pair)

```
bun run ../../tools/u2-mesh/gen-identities.ts \
  --count 2 \
  --label-prefix bringup \
  --out-dir /tmp/bringup
```

`bringup-01.json` → laptop. `bringup-02.json` → first Pi. `loopback` defaults to `false` which is correct now (we want real LAN multicast, not loopback echo).

**Step 3 — Flash one SD card with Armbian for Orange Pi Prime**

Latest stable from [armbian.com/orange-pi-prime](https://www.armbian.com/orange-pi-prime/) (pick the minimal CLI image, aarch64). Use `dd` or balenaEtcher to write to one of the existing skyminer microSDs (the original Skywire firmware will be overwritten).

```
sudo dd if=Armbian_*_Orangepiprime_*.img of=/dev/diskN bs=4M status=progress
```

**Step 4 — Boot the Pi in the skyminer**

Slot the SD card back into one of the SBCs, leave the enclosure wired up so it's on the internal PL-DGMK300 router LAN. Power it on. Armbian's first boot takes ~90s; it'll create a default user and force a root-password change.

Find the Pi's IP from the router (log into `192.168.0.1`, `admin`/`admin`, look at the DHCP client list) or from your laptop:

```
# From any host on the LAN segment:
nmap -sn 192.168.0.0/24
# Look for "Orange Pi" or a new MAC starting with 02:81:21 (Allwinner OUI).
```

Note: the skyminer's internal router is its own LAN segment. You'll need to either plug your laptop into the same switch OR put the laptop on the WAN side of the PL-DGMK300 and forward (simpler: plug into the same switch).

**Step 5 — SSH in, set initial password, then bring up mesh-node**

```
ssh root@<pi-ip>    # default password is on the Armbian download page; you'll be forced to change it
# Create a user when prompted, then exit root, become that user, sudo as needed.

# On your laptop (in another terminal), copy the three artifacts to the Pi:
scp zig-out/bin/mesh-node user@<pi-ip>:/tmp/
scp /tmp/bringup/bringup-02.json user@<pi-ip>:/tmp/mesh.json
scp ../../tools/u2-mesh/mesh-node.service user@<pi-ip>:/tmp/
scp ../../tools/u2-mesh/install-on-pi.sh user@<pi-ip>:/tmp/

# Back on the Pi:
sudo /tmp/install-on-pi.sh /tmp/mesh-node /tmp/mesh.json /tmp/mesh-node.service
```

The install script verifies the binary is aarch64, installs the binary to `/usr/local/bin/mesh-node`, drops the config at `/etc/semantos/mesh.json`, registers the systemd unit, starts it, and tails the journal. You should see `mesh-node up` + `TX heartbeat #0` immediately.

**Step 6 — Launch the laptop side**

```
# On your laptop, in semantos-brain/:
zig-out/bin/mesh-node --config /tmp/bringup/bringup-01.json --heartbeat-ms 2000
```

Within ~2 seconds you should see `RX heartbeat from bringup-02` lines on the laptop, and on the Pi's journal you should see `RX heartbeat from bringup-01`. If yes — the substrate is alive across actual silicon, across the actual switch.

**Step 7 — Confirm IPv6 multicast is the path**

On the Pi:

```
ip -6 maddr show          # confirms the mesh-node socket joined ff15::5e:1
ss -ulnp | grep mesh-node # confirms the UDP6 listener on port 47100
sudo tcpdump -i eth0 -nn 'ip6 multicast' -c 5
```

You should see IPv6 multicast frames going out and being received from the laptop's MAC.

### Bake script (8-Pi /loop body) — after the first card validates

Once step 7 of the First-Pi bring-up is green, the bake-all-8 flow is mechanical: same Armbian-for-Orange-Pi-Prime image, but use Armbian's `userpatches/customize-image.sh` hook to bake `mesh-node` + the systemd unit into the image, then generate 8 configs and inject per-card during flash. That's the `/loop` body — outer loop over `node-01.json` ... `node-08.json`, inner step prompts for the next SD card, flashes the customized image, mounts the rootfs (ext4, requires ext4-fuse on macOS or a Linux helper VM), drops the JSON to `/etc/semantos/mesh.json`, ejects.

This script will be written once the first Pi gossips happily — it depends on choices that are only obvious after first-Pi validation: hostname pattern, whether to disable onboard WiFi (probably yes — wired-only mesh), whether to bind `mesh-node` to `eth0` specifically vs `::`, and whether the PL-DGMK300 needs any IPv6-related toggling.

### Coexistence with bitcoin-shard-proxy

The bake script targets a single SD image carrying TWO systemd units — `mesh-node.service` (our substrate gossip) and `bitcoin-shard-proxy.service` (BSV tx fanout). They share the wire but not the namespace:

- mesh-node listens on `[ff15::5e:1]:47100` (configurable per node JSON).
- bitcoin-shard-proxy listens on its own ingress port (UDP/TCP) and emits to `FF05::B:<shard>` for data plane + `FF05::B:FFFA-FFFE` for control.

No port collision, no group collision. Each Pi runs both. The skyminer becomes a tiny incarnation of the IPv6-only + BSV-blockchain-on-multicast vision the IPv6 Forum is calling for in 2026.

## Architectural notes for reviewers

**Why a separate UDP slot in `event_loop.poll_fds` rather than two `poll()` calls.** The brief's reactor-isolation property is "a stalled connection of one type cannot starve the other." Single poll, one fd per surface, drain-until-WouldBlock is the cleanest realization. The slot is gated behind `attachUdpDispatcher()`; pure-TCP brain deployments are byte-identical to before.

**Why multicast (not pure unicast) is the right LAN default.** The brief is unicast (peers track each other via HEARTBEAT-updated `lastSeenAddr`). For an 8-Pi LAN cluster, unicast forces either an mDNS-style discovery layer or a static peer-address config. Multicast sidesteps both: every node joins `[ff15::5e:1]:47100`, broadcasts, everyone hears. Pre-shared per-sender broadcast secrets (from `gen-identities.ts`) handle authentication; the multicast group handles delivery.

**Why pre-shared keys are correct for v1.** Phase U.3 ships ECDH-derived secrets via the contacts cell-DAG. Until then, baking per-sender broadcast secrets into 8 SD cards is operationally simple (single-operator testbed), cryptographically equivalent at the dispatcher layer (HMAC framing is identical), and zero-rework when U.3 lands (just swap the `PeerSharedSecretLookup` implementation).

**IPv6 multicast constants live in `udp_dispatcher.zig`.** Zig 0.15's `std.posix` doesn't expose `IPV6_ADD_MEMBERSHIP` / `IPV6_MULTICAST_HOPS` / `IPV6_MULTICAST_LOOP` / `IPV6_V6ONLY`. The constants differ between Linux and Darwin; we branch on `builtin.os.tag`. If you port to Windows or BSD, add a new branch in the four `ipv6Foo()` helpers near the bottom of `udp_dispatcher.zig`.
