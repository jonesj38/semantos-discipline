# Semantos ¬∑ Sprint Wrap-up ¬∑ May 2026

---

## The Vision in One Paragraph

Semantos is a **sovereign node protocol** for AI-powered business apps. Every operator ‚Äî a tradie, a property manager, a consultant ‚Äî gets a Pask AI instance that knows their business and stores all their data as cryptographically-signed, content-addressed semantic cells. The AI can be hosted (multi-tenant on a shared brain box) or self-hosted (Raspberry Pi or bare metal). Either way, the operator holds the keys, owns the cells, and can exit at any time with a portable tarball. No accounts, no password resets, no custodial lock-in. Identity is a BRC-52 cert. Payments are BSV micropayment channels. The server runtime is a Zig binary called `brain`. The client is a Flutter app. The first product built on this stack is **Oddjobz** ‚Äî an intake-to-invoice system for Australian tradies.

## What Exists (the full technical inventory)

### The Brain (Zig server runtime)

`runtime/semantos-brain/` ‚Äî a single `brain` binary that handles everything server-side for one or many operators.

**Storage layer ‚Äî LMDB**
- `LmdbCellStore` ‚Äî content-addressed, 1KB semantic cells; key = `op_pkh(8B) ‚Äñ SHA256(cell)(32B)`. Every read/write/cursor is scoped to the operator's prefix. Cross-operator reads are structurally impossible.
- `LmdbPaskSnapshotStore` ‚Äî serialised Pask conversation graphs; key = `op_pkh(8B) ‚Äñ cert_id(16B) ‚Äñ version(8B)`. `rollbackTo` is per-operator.
- `LmdbHeaderStore`, `LmdbOutputStore`, `LmdbDerivationStateStore` ‚Äî BSV block headers and cell derivation scaffolding.

**Storage layer ‚Äî Postgres**
- `operators` table ‚Äî one row per hosted operator (`op_pkh` PK, `apex_domain`, `brain_domain`, `wrapped_dek`, `status`, `exiting_at`).
- 7 operator-scoped tables (`pask_node_view`, `pask_entailment`, `pask_stable_thread`, `session_chain`, `cells_lmdb_cache`, `action_cell_log`, `audit_log_cache`) all carry `op_pkh TEXT DEFAULT '0000000000000000'` and are governed by RLS policies that filter by `current_setting('semantos.op_pkh')`. `semantos_admin` bypasses. `semantos_brain` sees only its own rows.
- `op_metrics` materialised view ‚Äî per-operator cell count, byte storage, event rate, Pask graph size, session duration. Powers admin dashboard and billing hooks.
- `operator_exit(op_pkh)` SQL function ‚Äî atomic status transition + `exited_at` timestamp for the 30-day grace period.
- Migrations shipped: 017 (operators + RLS), 018 (metrics), 019 (exit), 020 (operator_profile JSONB).

**Identity + auth**
- BRC-52 cert chain validation: `wss_operator_auth.zig` ‚Äî SNI hostname ‚Üí `DomainMap.get` ‚Üí `op_pkh16` ‚Üí `X-Brain-Pubkey` header ‚Üí `certIdFromPubkey` ‚Üí per-operator `CertStore` ‚Üí `walkChain` (child‚Üíparent‚Üíroot, max depth 8). Delegated subkeys (BRC-42 children) accepted at any depth.
- `identity_certs.zig` ‚Äî cert store, BRC-52 record types, cert ID derivation.
- `bkds.zig` ‚Äî BSV key derivation, compressed secp256k1 pubkeys.

**Cryptographic isolation**
- Wrapped DEK: `wrapped_dek_store.zig` ‚Äî device generates a DEK, wraps it under a BRC-42-derived KEK, stores the opaque blob (`nonce‚Äñciphertext‚Äñtag` = 60 bytes) in `$data_dir/operators/<op_pkh16>/wrapped_dek`. Brain never sees the plaintext. Device unwraps on connect.
- `wallet.getWrappedDek` RPC ‚Äî returns the blob over the authenticated W7.4 WSS channel.

**Routing + TLS**
- `sni_domain_map.zig` ‚Äî JSON-backed map of `brain_domain ‚Üí op_pkh`. Loaded at startup, updated on operator provisioning.
- `domain_allowlist.zig` ‚Äî flat-file FQDN list for Caddy `on_demand_tls` ask endpoint.
- `caddy_ask_server.zig` ‚Äî `GET /caddy/ask?domain=<fqdn>` ‚Üí 200/403. Wired into `brain caddy-ask`.
- `caddy_template.zig` ‚Äî generates per-operator Caddy site blocks (apex ‚Üí intake bot, `brain.<domain>` ‚Üí WSS) plus global `on_demand_tls` block.
- SNI routing: `brain.<domain>` ‚Üí WSS on port 443. Caddy provisions certs via ACME on first connection. No wildcard cert, no manual cert management.

**Event streaming ‚Äî NATS JetStream**
- `nats_client.zig` ‚Äî TCP NATS client (publish, request-reply, stream create/delete, durable consumer create).
- `nats_event_producer.zig` ‚Äî emits FSM transitions to `op.<op_pkh16>.<hat>.fsm_transition`.
- Per-operator streams: `op_<op_pkh16>`, file-backed, 30-day retention, 10K msgs/subject cap.
- `nats_orphan_detector.zig` ‚Äî finds streams not in the active-operator list. Nightly systemd timer purges them.

**Operator lifecycle**
- `brain export-operator <op_pkh_hex> --output <path>` ‚Äî exports cells + Pask snapshot + manifest as TAR. K3-isolated, deterministic (LMDB key order).
- `brain exit-operator <op_pkh_hex>` ‚Äî export grace TAR ‚Üí delete LMDB cells ‚Üí delete Pask snapshots ‚Üí delete NATS stream ‚Üí checklist for rclone B2 + SQL exit function + Caddy removal.
- `brain provision-tenant`, `brain sni-map`, `brain domain-allow/disallow`, `brain wrapped-dek`, `brain caddy-ask` ‚Äî full operator lifecycle CLI.

**Semantos Sites 1.0**
- `operator_profile.zig` ‚Äî typed in-memory profile assembled from strategy cells (LBC, ICP, services, pricing).
- `operator_site_renderer.zig` ‚Äî HTML generation: nav, hero (h1 + lede + trust signals), services grid, how-it-works steps, pricing table, footer, analytics snippet. A/B variant resolution via session cookie.
- `operator_profile_loader.zig` ‚Äî reads `$data_dir/sites/<domain>/profile.json`, falls back to `defaultProfile`.
- `analytics_handler.zig` ‚Äî parses analytics events from POST `/api/v1/analytics`, appends to `analytics.jsonl`.
- `site_server.zig` ‚Äî `operator_home` route: SNI resolves domain ‚Üí load profile ‚Üí render HTML.
- CLI: `brain site-preview [<domain>] [--output <path>]`, `brain site-publish <domain> [--from <file>]`.
- `020_operator_profile.sql` ‚Äî `profile_lbc`, `profile_icp`, `profile_services`, `profile_pricing`, `site_status` columns on `operators` table.
- Wizard system prompt: 4-phase conversation to generate hero h1/lede, trust signals, service slugs, pricing.

**Application domain ‚Äî Oddjobz**
- 8 typed cells: job, quote, visit, invoice, customer, site, estimate, message.
- FSMs: job/quote/visit/invoice state machines with valid transitions.
- Jobs, customers, quotes, invoices, visits, leads, attachments ‚Äî all LMDB-backed stores.
- Intent cells: LLM-extracted entities ratified by the operator.
- Intent action router: maps ratified intents to job FSM transitions.
- `oddjobz_ratify_handler`, `oddjobz_query_handler`, `oddjobz_attention_handler`.

**Legacy ingest**

- Email extractor, attachment extractor (PDF byte-parse still a gap), OpenRouter LLM client.
- Pre-classifier ‚Üí thread collapser ‚Üí proposal store ‚Üí ratification orchestrator.
- `pask-bridge.ts` ‚Äî surfaces proposals to the brain for Pask integration.

## The Emerging Vision ‚Äî What This Is Actually Becoming

The technical work has crystallised something about what Semantos is.

**It's a protocol, not a product.** Oddjobz is an application built on the Semantos cell engine. Other applications (property management, clinic administration, inventory) could run the same brain binary with different extension cells and a different Pask grammar. The operator isolation (W7) means the hosting economics are generic ‚Äî any small-business vertical that generates modest cell volumes could be hosted on the same box.

**The cell engine is the differentiator.** Not the UI, not the AI models, not the hosting. The cell engine enforces linearity (a job can only be invoiced once), content-addressing (a cell's identity is its content, not a database row ID), and operator isolation (no cross-contamination, structural not policy). Those properties hold whether the cell is in LMDB on the VPS, in SQLite on the phone, or in a browser WASM module. One engine, three runtimes, identical semantics.

**Sovereignty is the pitch, not the tech.** "You own the keys, the data, the Pask graph, the domain. We host it efficiently. You can leave." That's a different commercial relationship than SaaS. The tarball export (W7.7) and the non-custodial architecture (no password reset, Plexus holds recovery) are product features, not just engineering decisions. They're what makes the pitch credible.

**The Pask layer is where the long-term value compounds.** The cell engine is the substrate. Pask is what accumulates context over time ‚Äî it's the AI that learns the operator's business, their customers, their patterns. As the Pask graph grows, the system gets better at extracting intent, routing actions, anticipating needs. The longer an operator uses it, the more valuable it becomes. That's retention and switching cost without lock-in, because the Pask graph is in the portable tarball.

**The horizon: operator federation.** Right now, each operator's Pask graph is strictly isolated. The next architectural horizon (not in scope for this sprint or the next) is controlled federation ‚Äî an operator can grant a specific capability to another operator (e.g. a contractor can ratify visits on behalf of the business owner, scoped by BRC-42 subkey delegation). The cell engine already supports AFFINE capability tokens for exactly this. The Pask multi-operator conversation layer is in the M7 milestones. This is how the single-tradie product becomes a small-team product without breaking the sovereignty model.
