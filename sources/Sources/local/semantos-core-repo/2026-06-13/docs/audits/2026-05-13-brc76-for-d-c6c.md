# BRC-76 Evaluation for D-C6c

**Date:** 2026-05-13
**Triggered by:** `docs/prd/UNIFICATION-ROADMAP.md` §11.6 Tier-3 reference — "BRC-76 (Graph Aware Sync Protocol) — possible reference for D-C6c contract suite — audit during D-C6c RED"
**Author:** Tier-C burst iter 6
**Inputs:** BRC-76 spec, §11.2 D-C6c deliverable definition

---

## TL;DR

**BRC-76 is adjacent but not sufficient for D-C6c.** Different paradigm: GASP is bidirectional reconciliation (bloom filter + recursive INV); D-C6c is topic-based publish-resolve. **Do not bind D-C6c to BRC-76.** D-C6c stays bound to BRC-22 / BRC-24 per §11.6.

**However**, BRC-76 is the right reference for a **separate future workstream**: catch-up sync after a peer has been offline. That problem hasn't been scoped yet but should land as a tracker note before it becomes someone's surprise reinvention.

---

## 1. What D-C6c requires

§11.2 specifies the contract test suite that every `NetworkAdapter` implementation must pass:

> **D-C6c (U6×C):** NetworkAdapter contract test suite (Phase-26D). Every implementation must pass: publish-then-resolve roundtrip, idempotent publish, ordered delivery within topic, backpressure under N×fanout. Reusable across MulticastAdapter, WsNodeAdapter, InMemoryAdapter.

Four properties:

| Property | What it means |
|---|---|
| publish-then-resolve roundtrip | After `publish(topic, frame)`, a subsequent `resolve(<frame's address>)` returns the frame |
| idempotent publish | Calling `publish(topic, frame)` twice produces the same result as calling it once |
| ordered delivery within topic | Subscribers receive frames in the order they were published, within a single topic |
| backpressure under N×fanout | When N subscribers slow consumption, the adapter signals back to producers rather than dropping silently or OOMing |

These are message-bus / pub-sub properties. The closest BRC standards are BRC-22 (Overlay Network Data Synchronization) for `publish` semantics and BRC-24 (Overlay Network Lookup Services) for `resolve` semantics — both of which §11.6 already binds D-C6c to.

---

## 2. What BRC-76 specifies

BRC-76, authored by Ragnar Friedman, is described in the spec itself as an "**exploratory graph synchronization proposal**" and explicitly states it is **not** the current wallet-toolbox protocol — that role belongs to BRC-40 for storage and BEEF for transaction exchange.

### Sync unit

**Transactions and their associated metadata**, not cells. The protocol traverses the transaction graph by following each transaction's inputs to its ancestors.

### Wire format

Two primary message constructs:

1. **Bloom filter** — Alice initiates with "a bloom filter containing all current spendable TXID+VOUTs as elements."
2. **INV (Inventory) message** — Bob responds with INV messages containing "Output, transaction, and associated merkle proof" plus metadata (labels, descriptions, timestamps) and "a list of VOUTs spent by its inputs and associated metadata hashes."

### Protocol shape

A **stateful bidirectional reconciliation**:

```
Alice → Bob:  bloom filter of "what I have"
Bob → Alice:  INV messages for transactions Alice's filter doesn't contain
Alice → Bob:  recursive request for any unknown inputs in those INVs
            (continues until Alice has the full ancestry she needs)
Role reversal:
Bob → Alice:  bloom filter of "what I have"
Alice → Bob:  INV messages for transactions Bob's filter doesn't contain
            (continues until Bob has the full ancestry he needs)
```

This is the same pattern as `git fetch` + `git push` — each side first describes its current state, then the other side fills the gaps.

### What BRC-76 explicitly does NOT specify

| Property | BRC-76 status |
|---|---|
| publish-then-resolve roundtrip | **No.** GASP doesn't separate "publish" from "resolve"; it's a single reconciliation flow. |
| idempotent semantics | **No.** No mention of idempotency for repeated operations. |
| ordered delivery within topic | **No.** No topic concept; no ordering guarantees. |
| backpressure under N×fanout | **No.** GASP is "declaratory" — "failures experienced by one party are not communicated to the other." Silent failure. |
| Integration with BRC-22 / BRC-24 | **No.** Only references BRC-40 to distinguish itself. |
| Wire-format details | **No.** Transport-agnostic; bloom filter and INV are logical messages. |

## 3. Side-by-side: BRC-76 vs D-C6c

| Property | D-C6c needs | BRC-76 provides |
|---|---|---|
| Communication shape | publish / subscribe / resolve over topics | bidirectional reconciliation between two parties |
| Sync unit | cells (1024 B content-addressed objects) | transactions + metadata |
| Initiator | producer pushes; consumer pulls (separate roles) | both parties act as initiators in turn |
| State machine | stateless per-message | stateful per-session (bloom → INV → recursive → role reverse) |
| Topics / channels | yes, with ordered delivery | no |
| Idempotency | required | unspecified |
| Backpressure | required | declaratory silent failure |
| Reference to BRC-22 / BRC-24 | inherits semantics | none |

The two specifications solve **different problems**. D-C6c is a message bus for ongoing cell propagation; BRC-76 is a catch-up reconciliation for transaction history.

---

## 4. Verdict

**D-C6c MUST NOT bind to BRC-76.** The properties D-C6c's contract suite enforces (idempotent publish, ordered delivery, backpressure) are explicitly absent from BRC-76. Binding would weaken D-C6c's contract to "whatever GASP gives us", which is silent failure under load.

**D-C6c stays bound to BRC-22 (Overlay Network Data Synchronization) for `publish` semantics and BRC-24 (Overlay Network Lookup Services) for `resolve` semantics**, per §11.6 Phase-26D's pre-existing binding. The contract test suite enforces the four properties on top of those BRC bases.

---

## 5. Where BRC-76 IS useful (separate workstream)

BRC-76's bidirectional reconciliation pattern is the right answer to a different question Semantos hasn't formalized yet:

> *When a peer has been offline for a period and reconnects, how does it catch up on the cells it missed without flooding the wire with every cell since the partition?*

This is the "git fetch" problem at the cell-graph level. The current substrate has no formal catch-up protocol — when a peer reconnects, the implicit assumption is that it'll start consuming the live stream and reconcile gaps via per-topic semantics. That works for short partitions but degrades for long ones.

Three scenarios that benefit from BRC-76-style reconciliation:

### Scenario A: Operator-to-operator federation catch-up

When a Phase-35B `WsNodeAdapter` reconnects after a network partition, both nodes need to converge on the shared dispatch-envelope cells. A bloom filter of "cells I have for this dispatch envelope" + recursive INV is a clean way to converge without each side dumping its entire history.

### Scenario B: Cross-region two-phase commit reconciliation

§11.2's D-W2 (cross-region two-phase commit) needs to reconcile entity hash chains across regions when a transfer crosses the boundary. If the destination region is briefly unreachable during the despawn-in-A → spawn-in-B sequence, the recovery path is a BRC-76-shaped exchange.

### Scenario C: FederatedPeerLocator (35B.3) registry sync

The future `FederatedPeerLocator` (operator-run registry) needs to converge on the peer endpoint set across multiple registry operators. BRC-76 is structurally close to what the registry needs — bloom of known endpoints + INV of new ones.

### What to do about it now

**Add a tracker note**, not a deliverable. Something like:

> **D-Sync-1 (DESIGN, future):** Catch-up reconciliation protocol for cells after partition. Reference: BRC-76 (Graph Aware Sync Protocol). Adapted from BRC-76's transaction-graph paradigm to cells-and-patches. Not blocking V1; lands when a real catch-up requirement surfaces (e.g. WsNodeAdapter reconnect testing, FederatedPeerLocator registry sync).

This keeps BRC-76 in the visible reference set without committing to implementation work that isn't currently needed.

---

## 6. Recommendation

1. **Do not bind D-C6c to BRC-76.** Confirm D-C6c stays bound to BRC-22 + BRC-24 per §11.6. No edit to UNIFICATION-ROADMAP §11.6 required.
2. **Add D-Sync-1 as a future deliverable note** to §11.6 (or to a new "Tier 3 deferred work" subsection). Reference BRC-76 as the protocol pattern.
3. **Track BRC-76 in §11.6's "Tier 3 — worth tracking" table** with the updated framing: *"BRC-76 (Graph Aware Sync Protocol) — pattern reference for future catch-up sync workstream (D-Sync-1); not applicable to D-C6c."*
