# BRC-120 vs BRC-105 — Axis G External Interface

**Date:** 2026-05-13
**Triggered by:** `docs/prd/UNIFICATION-ROADMAP.md` §11.6 **OD-BRC-2** — "BRC-120 vs BRC-105 for axis G. Pick one as canonical external interface."
**Author:** Tier-C burst iter 7
**Inputs:** BRC-105, BRC-120 (deep reads), §11.6 bindings

---

## TL;DR

**Pick BRC-120 as canonical external axis-G wire format.** Frozen spec, stateless, sovereign-node-fit. Keep BRC-105 as a **transitional / interop fallback** for V1 prototyping (reuses the existing `ts-sdk AuthFetch` reference implementation).

**Suggested phasing:**

1. **Phase G-1 (V1 prototyping):** Implement BRC-105 first, using the existing `ts-sdk AuthFetch` infrastructure. Operator-to-tenant payments work end-to-end against the existing BSV ecosystem.
2. **Phase G-2 (V2 federation):** Add BRC-120 as a parallel external interface for stateless cross-tenant payments. Both protocols ship; operator config selects which is offered at edge.
3. **Phase G-3 (deprecation):** When the x402 v1.0 reference implementation matures and operators have migrated, deprecate BRC-105.

This staged approach avoids the "no migration path" risk noted in BRC-120 (which makes no mention of BRC-105 compatibility) by treating the two as independent endpoints rather than versions of the same protocol.

## 2. Why BRC-120 is the right canonical choice

### 2.1 Frozen status

BRC-120 (via x402 v1.0) has **Specification Status: Frozen**. BRC-105 is explicitly still being extended — three later BRCs (118, 120, 121) "extend BRC-105 without replacing it unless they are being adopted explicitly."

For an external wire format that operator nodes expose at their edge — and that other operators / clients depend on for payment-gated access — **stability matters more than feature richness.** Frozen wins.

### 2.2 Statelessness aligns with sovereign-node architecture

The substrate paper's sovereign-node story (per `docs/prd/UNIFICATION-ROADMAP.md` §0 north star) is "voice to economic execution on independent nodes." Operator nodes don't share session state across organizational boundaries.

BRC-105's BRC-103 session establishment is *good* for intra-tenant use (single operator, multiple internal services) but *wrong* for cross-tenant federation:

- Cross-tenant payment = "operator A's tenant pays operator B for a service"
- BRC-103 establishment between A and B implies a sustained mutual-auth session
- Per `runtime/ws-node-adapter/README.md`, Phase-35B already has its own license-handshake mechanism — adding BRC-103 sessions on top duplicates the auth state

BRC-120's stateless model avoids this: every request carries its own proof, no session state on either side. The federation transport (Phase-35B) handles peer identity at the connection layer; payments handle settlement at the per-request layer; the two don't entangle.

### 2.3 RFC 8785 canonical JSON

For payments that anchor on-chain, the serialization needs to be byte-identical across implementations. RFC 8785 (JSON Canonicalization Scheme) is the right standard — same input bytes produces same hash, regardless of which library serialized it.

BRC-105 doesn't specify a canonical serialization. In practice this means a server signing one canonical form and a client signing a different canonical form will fail signature checks despite producing semantically-identical JSON. Subtle bug class.

### 2.4 UTXO-as-nonce is the BSV-native replay primitive

BRC-105's nonce is a plain header value. BRC-120 uses UTXO consumption itself as the replay primitive — a payment can only be settled once because the UTXO can only be spent once. This is the same property that makes capability tokens replay-safe (per §11.6's D-Dcap-engine binding to BRC-108 + BRC-115).

Using the same primitive for capability AND payment replay protection is architecturally consistent.

## 3. Why BRC-105 is the right *transitional* choice

### 3.1 Reference implementation exists

BRC-105 cites `ts-sdk AuthFetch` as the reference impl. BRC-120 cites no implementation. For V1 prototyping, an existing TypeScript reference is operationally cheaper than implementing x402.md from scratch.

### 3.2 Documented inline

BRC-105 documents its header contract and JSON body inline. BRC-120 punts to x402.md, which lives in a separate repository. Implementers need to read both. For a V1 doc-burst-level handoff, single-document specs are easier to ship.

### 3.3 BRC-103 reuse

The federation transport already wires BRC-103 via license-handshake (Phase-35B). Intra-tenant calls can reuse that session for payment without negotiating a separate proof path. **For internal-operator paid services** (e.g., operator A's internal tenant paying operator A's internal service), BRC-105 is a clean fit.

### 3.4 Ecosystem support

If any existing BSV ecosystem services already accept BRC-105 payments (and the `ts-sdk AuthFetch` lineage suggests the broader BSV TypeScript ecosystem leans this way), implementing it first gives interop with shipped third-party services. BRC-120's ecosystem is younger.

## 4. The staged adoption proposal

Three phases:

### G-1: BRC-105 for intra-tenant / V1 prototyping

**Scope:** Operator A's tenant paying Operator A's internal service. Single-tenant test deployments.

**Implementation:** Reuse `ts-sdk AuthFetch`. Wire the 6 BRC-105 headers into the operator's HTTP service framework. Use the existing BRC-103 session (already established by license-handshake at the federation layer or directly per pair).

**Acceptance:** Test deployment of one operator with a paid internal service (e.g., `/api/v1/llm-completion` requiring 100 sats per call). 402 → payment → response works end-to-end.

**Cost:** ~1 week if `ts-sdk AuthFetch` is reused as-is. ~3 weeks if it needs adaptation.

### G-2: BRC-120 parallel external interface

**Scope:** Cross-tenant federation. Operator A's tenant paying Operator B's service. Stateless, no BRC-103 session.

**Implementation:** Implement x402 v1.0 spec from `spec/x402.md`. Add as parallel HTTP endpoint at operator edge. Operator config selects which protocol is offered (or both).

**Acceptance:** Test deployment with two operators where tenant of A pays for service from B without an active BRC-103 session between A and B.

**Cost:** ~3 weeks (implement x402 from spec; conform tests; integration test across two operators).

### G-3: Deprecate BRC-105 (long-term)

**Scope:** Once BRC-120 reference impls mature in the wider ecosystem and operator migrations complete, deprecate BRC-105 at operator edges.

**Implementation:** Remove BRC-105 endpoint; keep BRC-105 client-side for talking to older third-party services.

**Acceptance:** No operator-edge BRC-105 service active. BRC-120 is the only canonical external axis-G interface.

**Cost:** Operational, not engineering. Depends on ecosystem readiness.

## 5. What this implies for the unification roadmap

### Edit to §11.6 OD-BRC-2

Replace:

> **OD-BRC-2: BRC-120 vs BRC-105 for axis G.** BRC-120 is stateless and aligns with sovereign-node story; BRC-105 has more ecosystem and richer header set. Pick one as canonical external interface. Lands during D-G1.

With:

> **OD-BRC-2 (resolved 2026-05-13, audit `docs/audits/2026-05-13-brc120-vs-brc105.md`):** Axis G canonical external wire format is **BRC-120** (frozen x402 v1.0, stateless, RFC 8785 deterministic JSON, UTXO-as-nonce replay). BRC-105 is the **transitional / V1 implementation** path — reuses `ts-sdk AuthFetch`, requires BRC-103 session (acceptable intra-tenant). Staged phasing: G-1 BRC-105 prototyping → G-2 BRC-120 parallel → G-3 BRC-105 deprecation.

### New tracker entries

Lift the three phases into §11.2 deliverables:

- **D-G1a:** BRC-105 intra-tenant prototyping using `ts-sdk AuthFetch`. Phase G-1.
- **D-G1b:** BRC-120 (x402 v1.0) parallel external interface for cross-tenant federation. Phase G-2.
- **D-G1c (DEFER):** BRC-105 deprecation at operator edges. Phase G-3. Conditional on ecosystem readiness; not gated.

These supersede the §11.2 `Axis G` line item that simply names "BRC-120 primary, BRC-105 fallback" without defining the phasing.

### Internal channel layer (MFP) is unaffected

Both BRC-105 and BRC-120 are *single-shot per request* external interfaces. **Recurring billing remains MFP's job** (the internal channel-FSM in `extensions/metering/`). The external interface is the per-tick payment carrier; MFP is the channel that batches ticks into a settlement.

This is the load-bearing distinction §11.6 already noted: MFP is the internal channel optimization; BRC-120/105 is the per-tick external interface.

## 6. Open questions still

These don't block the BRC-120 vs BRC-105 decision but warrant tracker entries:

1. **x402 v1.0 reference implementation availability.** BRC-120 cites the frozen spec but names no reference impl. Track when an authoritative TypeScript / Zig implementation lands.
2. **RFC 8785 dependency.** Implementations need a canonical JSON library. Verify available libraries: `canonical-json` (npm), `json-canonical` (rust), Zig — likely need to roll our own.
3. **UTXO-as-nonce mechanics.** BRC-120 punts to x402.md for "any checks the spec mandates around reuse of proofs or spends." Phase G-2 implementation needs to read x402.md cover-to-cover, not just BRC-120.
4. **Cross-protocol semantic equivalence.** If an operator offers both BRC-105 and BRC-120 in parallel (during phase G-2), the same paid service must produce the same outcome for either protocol. Define a test matrix.
