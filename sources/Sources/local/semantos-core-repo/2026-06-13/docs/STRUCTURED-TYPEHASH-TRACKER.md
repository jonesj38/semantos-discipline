## Why this exists

The 32-byte `typeHash` field at cell-header offset 30 is currently computed by **three different rules** in three different code paths (MNCA dotted name, Oddjobz colon-triple, extension script `category || name`), all producing **opaque flat hashes** that no relay, LMDB scan, or projection layer can use without dereferencing back to a string.

The decision record `STRUCTURED-TYPEHASH-CANONICAL.md` resolved the architecture:

- **Construction:** four concatenated 8-byte truncated SHA-256 segments → routes hierarchically via prefix match
- **Source of truth:** cartridge manifest, not Zig and not TS
- **Algorithm in binary, identities in manifests** — opt-in by shape
- **Reserved wildcard:** raw `0x00 × 8` prefix as explicit promiscuous-route sentinel
- **MNCA reclassified** as substrate cartridge with source-cartridge embedded in `segment1` for routing locality

This tracker drives the five-step migration to land it.
