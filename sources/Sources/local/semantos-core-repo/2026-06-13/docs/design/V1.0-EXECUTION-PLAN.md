## 0. Mission

A working sovereign node — one Zig binary on one VPS — that simultaneously:

1. Serves `oddjobtodd.info` as a website (replacing Vercel)
2. Hosts the operator's BRC-100 wallet (replacing Firebase auth + MetaMask)
3. Gates content via HTTP 402 / identity-cert challenges (replacing Auth0 + Stripe)
4. Holds the operator's customer / job / quote / calendar universe as substrate cells (replacing Postgres-as-source-of-truth)
5. Surfaces what matters next on Helm via an inferred ranked attention feed
6. Accepts voice commands via a `do | find | talk` modal grammar
7. Pulls the operator's history from Gmail / Meta / WhatsApp / Google Calendar / Xero through a Paskian ratification loop until the legacy apps are read-only mirrors

OJT (`oddjobtodd.info`, an operating handyman business on the Sunshine Coast) is the
inaugural deployment. The textbook's running worked example is the operator's
business; the substrate's first production case is its own canonical demo.

The production VPS is reachable via:

```bash
ssh rbs
```

That alias is configured in the operator's `~/.ssh/config` and resolves to the host
that serves `oddjobtodd.info`. Use it for:

keeps the existing Next.js bot online throughout; BRAIN lands on a sibling port until

### Stage 0 — Schema Audit (~1 hour)

**Goal**: confirm the Postgres schema on the production VPS matches what the Drizzle
migrations declare, and reconcile any drift before WSITE phase 3 (substrate cutover).

**Prerequisites**: `ssh rbs` works.

**Steps**:

1. `ssh rbs 'pg_dump --schema-only $DATABASE_URL'` against the production DB.
   Save the output to `/tmp/ojt-prod-schema.sql`.

2. Locally, generate the expected schema from the Drizzle migrations:
   ```bash
   cd /Users/toddprice/projects/oddjobtodd
   pnpm install
   pnpm db:generate                    # produces drizzle/meta/snapshot
   ```
   Or — quicker — concatenate the 10 migrations as the de facto schema source:
   ```bash
   cat oddjobtodd/drizzle/0000_*.sql oddjobtodd/drizzle/0001_*.sql ... > /tmp/ojt-expected-schema.sql
   ```

3. Diff:
   ```bash
   diff -u /tmp/ojt-expected-schema.sql /tmp/ojt-prod-schema.sql > /tmp/ojt-schema-drift.diff
   ```

4. Categorise drift into:
   - **Forward-compatible additions** — prod has columns/tables not in the
     migrations (operator hand-added). Capture as a synthetic migration
     `oddjobtodd/drizzle/00NN_recover_prod_drift.sql` so future migrations
     are correct.
   - **Schema regressions** — migrations declare things prod doesn't have
     (failed migration). Decide whether to apply the missing migration on
     prod or revert the migration in the repo.
   - **Acceptable divergence** — prod-only test artefacts, etc. Document
     and ignore.

**Acceptance gate**: a single canonical schema is declared. Both production VPS
Postgres and any future deployment will run the same Drizzle migration set
through to head. The drift diff is committed to `docs/design/post-mortems/
OJT-SCHEMA-DRIFT-2026-04.md` (create the dir if needed).

**Output for next stages**: clean schema is the source of truth for stage 4
(substrate cutover) — every cell shape that maps onto a Postgres row needs a
declared substrate cell-type. The schema diff is one half of that mapping.

### Stage 1 — BRAIN on the VPS (~10 days, see BRAIN §5 sizing)

**Goal**: a Zig binary on the OJT VPS that loads the wallet engine + headers
verifier WASM modules, exposes a REPL over SSH/WSS/HTTP, and persists wallet state
to lmdb. The Next.js bot keeps running unchanged.

**Doc**: [`WALLET-SHELL-VPS-SUBSTRATE.md`](../design/WALLET-SHELL-VPS-SUBSTRATE.md) §3 phases Brain 1–Brain 6.

**Prerequisites**:
- WH1 + WH2 already landed (cell-engine PoW verifier + header store; see commits
  `1542738` and `cf62e46` on main).
- WA1–WA4 in flight (OutputStore + internalizeAction + recovery scan).
- Stage 0 complete.

**Steps**:

1. **Brain 1 (host shell + wasmtime)**: implement `runtime/shell/src/host-shell.zig`
   embedding wasmtime via its C API, hash-pinned module loader, lifecycle CLI
   (`semantos init|start|stop|status|hash|exit`).
2. **Brain 2 (broker)**: `runtime/shell/src/host-broker.zig` mediating every
   wasmtime host import; lmdb storage backings; audit log; module-isolation
   policy enforcement.
3. **Brain 3 (TUI REPL)**: `runtime/shell/src/repl-tui.zig` with the v0.1 command
   set (identity, balance, send, anchor, ledger, recover, policy, sync, status,
   exit). History persistence; tab completion.
4. **Brain 4 (remote surfaces)**: WSS endpoint at `:8090/wallet`, HTTP REPL at
   `/api/v1/repl`, SSH-as-default. Caddy front for TLS.
5. **Brain 5 (LLM adapter, optional)**: layered above the REPL; off by default;
   prepare the schema for VS to consume in stage 6.
6. **Brain 6 (deployment recipes)**: Docker, systemd, NixOS, Ansible playbooks;
   `deploy/README.md`.

**On the VPS**:

```bash
ssh rbs
sudo systemctl status semantos-ojt          # confirm Next.js bot is running
# Install BRAIN binary into /usr/local/bin and systemd unit into /etc/systemd/system
# (the BRAIN unit `semantos-shell.service` lives in runtime/shell/systemd/)
sudo systemctl enable --now semantos-shell
sudo systemctl status semantos-shell
ssh -p 22 rbs semantos status               # smoke test
```

The new unit binds to a *different* port from Caddy's existing Next.js front.
No traffic is shifted yet.

**Acceptance gate** (per [BRAIN §7](../design/WALLET-SHELL-VPS-SUBSTRATE.md#7-acceptance-criteria)):

- `semantos start` boots cleanly, both modules load with verified hashes,
  `semantos status` reports healthy.
- Hash-pinning is enforced — tampered WASM file aborts startup.
- Audit log accumulates for every host import call.
- Bundle size: shell binary + both WASM modules + wasmtime ≤ 25MB total.
- Memory footprint: idle wallet < 256MB RAM; under load < 512MB.
- WSS endpoint accessible from a remote browser with bearer-token auth.

### Stage 2 — WSITE Admin Routes (~7-8 days, see WSITE §7 sizing)

**Goal**: the four Next.js admin pages (`/admin/leads`, `/admin/chat`,
`/admin/calendar`, `/admin/import-job`) are reproduced as WSITE-served routes on
the VPS and gated by BRC-100 identity-cert auth (operator's own cert as the only
issuer). Customer-facing routes (`/`, `/api/chat`, etc.) stay on Next.js for
this stage.

**Doc**: [`WALLET-SITE-AS-SOVEREIGN-NODE.md`](../design/WALLET-SITE-AS-SOVEREIGN-NODE.md) §3 phases WSITE1–WSITE5; §11 *Inaugural Deployment*.

**Prerequisites**: stage 1.

**Steps**:

1. **WSITE1 (config schema)**: author `~/.semantos/sites/oddjobtodd.info/site.toml`
   on the VPS declaring the existing Next.js routes as `type = "static"`
   passthroughs; admin routes scaffolded but `enabled = false` until step 4.
2. **WSITE2 (content serving)**: bring up `runtime/shell/src/site-server.zig`
   on the VPS at `:8080`; Caddy routes `/admin/*` to WSITE, everything else
   stays on Next.js. Static admin pages (login, the bare HTML hulls of the
   four admin routes) ship as static files.
3. **WSITE3 (auth + 402 + cert)**: implement `runtime/shell/src/auth-handler.zig`.
   Issue 401 challenges with `X-Semantos-Challenge`, accept `/auth/callback`
   POSTs, set first-party session cookies. Configure
   `identity_issuers = ["self"]` in `site.toml` so only the operator's own
   cert authenticates.
4. **Mobile auth roundtrip**: with WSITE3 live, the operator does the
   redirect dance once on his phone (`MOBILE-AUTH-FLOW` §3). His preferred
   wallet origin is `wallet.oddjobtodd.info` — same VPS — so the whole
   round-trip is local.
5. **Dynamic handlers**: replace each admin page's static hull with a WASM
   dynamic handler that calls the substrate VFS for live data
   (`type = "dynamic"`, `handler = "<route>.wasm"`). The handler reads
   from Postgres (read-cache) and from substrate cells where they exist.
6. **WSITE5 (admin REPL)**: REPL verbs `lead list`, `lead claim <id>`,
   `lead close <id> --outcome <X>`, `calendar slot <day> <time>`, etc.
   These are the existing admin panel actions exposed through the typed
   REPL — VS will pick them up in stage 6.

**Acceptance gate** (WSITE §9 + §11):

- `https://oddjobtodd.info/admin/leads` returns 401 with the correct challenge
  headers when accessed without a session cookie.
- The operator's BRC-100 redirect roundtrip succeeds on his phone.
- All four admin routes render with live data identical to the Vercel preview.
- Site config mutations are signed cells (auditable via REPL `site config history`).
- Vercel preview is still live; the new VPS-served admin can be reverted by
  dropping the WSITE entry in Caddy.

### Stage 3 — Auth Cutover Verification (~1 day)

**Goal**: cleanly retire Firebase auth. The operator's BRC-100 flow is the only
path to admin.

**Steps**:

1. With the WSITE-served admin running parallel to Vercel for at least 7 days,
   verify the operator has not had to fall back to Vercel for any operational
   need.
2. Disable Firebase auth on the production Vercel deployment (`vercel env rm
   NEXT_PUBLIC_FIREBASE_*`).
3. Update Caddy to no longer route any `/admin/*` traffic to Vercel — drop
   the rule.
4. Confirm `oddjobtodd.info/admin/login` (Next.js still serves the bare
   route, but redirects to the WSITE auth flow) takes the operator to the
   wallet origin redirect.
5. Close the Firebase project (or leave it dormant — costs nothing if
   unused).

**Acceptance gate**: no admin traffic touches Firebase or Vercel for 7 days.

### Stage 4 — Substrate-Truth Cutover (~5 days)

**Goal**: customer-state-mutating routes (`/api/chat`, `/api/v2/messages`,
`/api/v3/chat`) write to the substrate cell store *first*; Postgres writes
happen as a denormalized projection from the cell mutation. Postgres becomes
a read-cache; the cell store becomes truth.

**Prerequisites**: stages 1, 2, 0 (schema audit informs which Postgres tables
correspond to which substrate cell-types).

**Steps**:

1. Declare cell-type schemas for every substrate-relevant Postgres table.
   Source: [`oddjobtodd/src/lib/db/schema.ts`](../../../oddjobtodd/src/lib/db/schema.ts) and the verticals
   schemas under [`oddjobtodd/src/lib/semantos-kernel/`](../../../oddjobtodd/src/lib/semantos-kernel/).
   The trades-vertical schema is already substrate-aligned —
   `schema.trades.ts` was authored against `schema.core.ts`. The work is
   bringing the core Drizzle schemas (`schema.ts`) into the substrate
   cell-type registry.

2. Implement a **dual-write adapter** in
   `oddjobtodd/src/lib/semantos-kernel/dual-write.ts`. Each Postgres
   write call passes through:
   - Build SIR program from the call
   - Pack as substrate cell (signed under operator's hat)
   - Persist via BRAIN host_persist
   - Then write Postgres row as denormalized projection from the cell
   - On any failure, rollback both

3. Run the dual-write in shadow mode for a week — every Postgres write also
   produces a substrate cell, but the Postgres row is still authoritative.
   Compare cell-state vs Postgres-state nightly; reconcile divergences.

4. Flip authority — substrate cells become the source of truth, Postgres
   reads still serve the UI for performance, Postgres writes happen as
   downstream projections.

5. Backfill historical Postgres rows as substrate cells (one-time batch
   job; idempotent; produces a synthetic legacy provenance pointer per
   cell).

**Acceptance gate**:

- Every customer-state-mutating API route produces a substrate cell that
  is the source of truth for that mutation.
- Reading from Postgres returns the same logical state as reading from
  the substrate VFS (verified by a daily reconciliation job).
- A backfill of historical Postgres rows produces substrate cells for
  100% of rows; cells carry the synthetic legacy provenance.

### Stage 5 — Legacy Ingest (~13 days, parallelisable from stage 2)

**Goal**: pull the operator's pre-substrate customer history from Gmail / Meta /
WhatsApp / Google Calendar / Xero into substrate cells via the Paskian
ratification loop.

**Doc**: [`WALLET-LEGACY-INGEST.md`](../design/WALLET-LEGACY-INGEST.md) §3 LI1–LI6.

**Prerequisites**: Brain 3 (REPL exists) + BRAIN HTTP surface (OAuth callback). Stage 2
strongly recommended (Helm right-panel ratification UX is a WSITE-served route).

**Steps**:

1. **LI1**: provider adapter interface, OAuth flow, encrypted-at-rest token
   store. The OAuth callback re-uses [`MOBILE-AUTH-FLOW`](../design/WALLET-MOBILE-AUTH-FLOW.md) §10.2's
   `purpose=oauth_grant` machinery.
2. **LI2**: Gmail adapter end-to-end. Operator runs `semantos legacy connect
   gmail`, OAuths on his phone, runs `semantos legacy ingest gmail
   --since 2020-01-01`, watches raw items flow in (encrypted in lmdb at
   `~/.semantos/legacy-ingest/gmail/`).
3. **LI3**: extractor over raw items → SIR proposals. Reuses
   `oddjobtodd/src/lib/ai/extractors/extractionSchema.ts` and
   `oddjobtodd/src/lib/ai/prompts/extractionPrompt.ts`. Per-content-type
   prompts (email/rfc822 first).
4. **LI4**: ratification queue — REPL verbs (`legacy review`, `legacy ratify
   <id>`, `legacy correct <id>`, `legacy reject <id>`), Helm right-panel
   surface, correction-feedback few-shot retraining.
5. **LI5**: convergence policy + cross-provider deduplication +
   reply-from-substrate (initially WhatsApp Cloud, later G-Cal).
6. **LI6**: parallel adapters — Meta Pages / Lead Ads, WhatsApp Cloud,
   Google Calendar, Xero.

**Acceptance gate** (LI §7):

- All five priority providers connect via OAuth from the operator's phone.
- ≥ 5,000 legacy items from ≥ 3 providers ingested into the substrate.
- ≥ 1,000 ratified into substrate cells.
- Reply-from-substrate verified end-to-end on WhatsApp.
- `legacy status` shows deprecation progress per provider.

### Stage 6 — Voice Shell (~6.5 days, parallelisable with stage 5)

**Goal**: operator presses mic on Helm, says "let me see Mrs Henderson's job,"
the parsed command renders the result in the centre panel. State-mutating
commands (`do | …`) prompt for explicit confirmation via the existing
`CommandApprovalCard`.

**Doc**: [`WALLET-VOICE-SHELL-GRAMMAR.md`](../design/WALLET-VOICE-SHELL-GRAMMAR.md) §3 VS1–VS5.

**Prerequisites**: Brain 3 (typed REPL). Phase 38G (`VoiceInput.tsx` +
`CommandApprovalCard.tsx`) is already merged in
[`apps/loom-react/src/helm/`](../../apps/loom-react/src/helm/) — VS extends it with the
`do | find | talk` modal layer rather than building from scratch.

**Steps**:

1. **VS1**: `runtime/shell/src/voice-grammar.ts` — modal-verb grammar +
   dispatch table from `(modal, what)` → concrete shell verb.
2. **VS2**: `runtime/shell/src/voice-parser-llm.ts` — LLM-as-parser-aid
   wrapping Brain 5's adapter; strict schema validation on output;
   confidence calibration.
3. **VS3**: extend `apps/loom-react/src/helm/VoiceInput.tsx` to wire its
   `onUtterance` through the new parser; render the parsed
   `VoiceCommand` preview; route confirmed commands through the existing
   `useShellDispatch` hook.
4. **VS4**: principal/object resolution + clarification dialogue UI in
   Helm (extend `CommandApprovalCard.tsx` with multi-candidate variant).
5. **VS5**: correction-feedback loop — every edit produces a parser-
   correction cell; few-shot retrieval at parse time.

**Acceptance gate** (VS §7):

- Voice end-to-end latency ≤ 2s on 4G.
- Confirm rate on routine utterances ≥ 90% after a week of corrections.
- Local-only mode (`backend = "local-whisper"` + `backend = "local-llama"`)
  works end-to-end with no third-party calls.
- State-mutating `do` commands always prompt; `find` / routine `talk`
  configurable to auto-confirm.

### Stage 7 — Attention Surface 39B+ (~7.5 days, parallelisable)

**Goal**: the right-panel attention feed becomes inferred and learning, not
deterministic with hard-coded weights. New signal sources (weather, Surfline,

**Doc**: [`HELM-ATTENTION-SURFACE.md`](../design/HELM-ATTENTION-SURFACE.md) §3 AS1–AS5.

**Prerequisites**: Phase 39A is already in tree (`AttentionEngine` +
`AttentionSurface.tsx`). Stage 5 (LI) provides the `legacy-ingest.proposal.created`
event stream that AS4 consumes. Stage 6 (VS) provides `find | next` for the
"what's next" voice surface in AS5.

**Steps**:

1. **AS1**: instrument `AttentionSurface.tsx`, add pin / suppress UX, persist
   typed `AttentionInteraction` events as signed cells.
2. **AS2**: `AttentionWeightLearner.ts` reads telemetry, drifts the five
   factor weights per operator + context profile (field / desk / night).
3. **AS3**: `~/.semantos/attention-rules.toml` + REPL verbs
   (`attention pin|suppress|must-show`).
4. **AS4**: `runtime/services/src/services/signals/{weather,surfline,
   legacy-ingest,capability}.ts` plug into the engine via the
   `AttentionSignalSource` interface.
5. **AS5**: mobile push subscription wired to immediate items;
   `find | next` voice surface; SMS fallback via the existing Twilio
   integration; quiet hours.

**Acceptance gate** (AS §7):

- Operator runs the system for a week; weights demonstrably drift in line
  with interaction patterns; `attention status` is informative.
- Weather + Surfline + legacy + capability signals are visible in the
  right panel.
- Push fires within 30s of an immediate item; voice "what's next" returns
  coherent summary.

### Stage 8 — Vercel Deprecation (~2 days)

**Goal**: the Zig binary on the VPS is the only thing serving `oddjobtodd.info`.
Vercel goes dark. The seven external dependencies pre-migration (Vercel,
Firebase, Postgres-as-truth, Upstash, Anthropic, Twilio, Vercel Blob) collapse to
Anthropic + Twilio (retained as utility) plus the substrate.

**Steps**:

1. Confirm every Vercel-served route has a WSITE equivalent that has been live
   for ≥ 30 days without a fallback to Vercel.
2. Caddy stops routing anything to Vercel.
3. DNS unchanged (still points at the VPS IP).
4. Vercel project paused (don't delete — keep for incident archaeology).
5. Vercel Blob storage migrated: any blobs the substrate now references
   directly via UHRP; legacy blobs stay accessible through a one-way mirror
   pointing at the same Vercel-Blob URLs (read-only) until the LI cutover
   produces substrate-native equivalents.
6. Final cost-baseline report: hosting cost / month before vs after.

**Acceptance gate**: no traffic to `*.vercel.app`; no Firebase API calls in 30d;
substrate is the source of truth for every customer mutation.

### 4.4 Loom vs Helm vs Shell

A naming distinction that matters when reading the pack:

- **Loom** — the renderer-agnostic service-layer coordination bus. Lives at
  `runtime/services/src/services/loom/`. Holds the warp threads (type system,
  taxonomy, identity, governance) under tension. Atoms + reducer + handlers
  + effects + visibility-rules. Substrate component U11. *Not a UI.*
- **Helm** — the visual three-panel React workbench, an adapter composing
  with Loom. Lives at `apps/loom-react/src/helm/`. Renders LoomState; produces
  LoomActions from operator gestures.
- **Shell** — the typed REPL / voice command path, a peer adapter to Helm.
  Lives at `runtime/shell/`. Parses operator input into typed verbs;
  produces LoomActions; same composition pattern as Helm with a different
  modality.
- **`loom-react`** (monospace) — the React package directory housing Helm.
  The "loom" in this name is historical and predates the Loom-as-service-
  layer naming; the path overlaps unfortunately with the Loom concept but
  is distinct from it. The package will be renamed `helm-react` when the
  integration debt is paid.

The architectural picture is: Loom is the warp; Helm and Shell are two
shuttles thrown through it. When wiring components, **state-holding services
go in Loom**; **rendering goes in Helm**; **input parsing goes in Shell**. A
prior version of the canon glossary listed Loom as an alias of Helm — that
conflation was corrected on 2026-04-27. See chapter 17b of the textbook
(`docs/textbook/17b-the-loom-service-layer-warp.md`) and
`docs/design/LOOM-SELF-HOSTING.md` for the architectural picture.

The practical implication for stage 6 (Voice Shell) and stage 7 (Attention
Surface 39B+): the AttentionEngine and the voice parser belong **in Loom and
Shell respectively**, not in Helm. Helm renders their output. Sessions
working on those stages should put new state-coordination code in
`runtime/services/src/services/` (Loom) or `runtime/shell/src/` (Shell) and
should add only render-layer changes to `apps/loom-react/src/helm/`.

### 4.5 Conventions

- Use Bun where possible (the workspace's `engines.node = ">=18"` and
  `packageManager = "pnpm@10.9.0"`, but the Zig and TS hot paths are
  Bun-callable per `runtime/shell/package.json` exports).
- Hash-pinning policy: every WASM module loaded by BRAIN has its SHA-256 in
  `core/cell-engine/WASM-MANIFEST.json`. New modules added: update the
  manifest in the same commit.
- Commit boundaries: one PR per phase deliverable per the boundary plans
  in each spec doc (e.g. BRAIN §6, WSITE §8, LI §6, VS §6, AS §6).
- Tests gate every merge: `bun test` for TS, `zig build test` for Zig,
  `pnpm -w check` for the workspace types.
- Audit log: every cross-module / cross-process event should be a signed
  cell or an audit-log entry. `host_audit` in Brain 2 is the canonical path.
